# Cordn Coordinator and Identity Model

- Status: Draft

## Abstract

This document defines the coordinator-facing baseline protocol model for [`cordn`](package.json). It specifies the role of coordinators, the role of clients, group-scoped message ordering, public fetch progression, and the initial identity-binding model used for MLS KeyPackages. The goal is to define a minimal, interoperable coordinator protocol surface that preserves clear trust boundaries while avoiding unnecessary specialization of coordinator roles.

In `cordn`, coordinators provide a uniform protocol surface. A group may operationally choose one coordinator as its current write target, but coordinators do not have distinct protocol roles such as primary-only writers or replica-only readers. Message ordering and pagination are scoped per group using monotonic cursors maintained by each coordinator. These cursors are ordering and fetch primitives, not stable message identities.

This document also defines how `cordn` binds stable Nostr identities to MLS BasicCredential identities for KeyPackage publication and retrieval. The design requires a signed KeyPackage publication payload that clients can verify independently, reducing the amount of trust placed in any single coordinator while preserving a simple coordinator-based distribution model.

## Specification

### 1. Overview

`cordn` defines a coordinator-assisted MLS delivery model.

- Coordinators expose a uniform protocol surface.
- Clients create and process MLS objects locally.
- Coordinators store and serve published KeyPackages, pending Welcomes, and ordered per-group message streams.
- Clients are responsible for MLS validation and for verifying identity bindings on coordinator-served publication payloads.

In `cordn`, coordinators are exposed as ContextVM servers. This contextualizes transport-derived caller identity, signed publication payload handling, and the coordinator tool surface used by clients.

This document defines only the common coordinator model and initial identity conventions. Group metadata is defined separately in [`spec/01.md`](spec/01.md).

### 2. Coordinator Role

All coordinators implement the same protocol surface.

- `cordn` does not define different protocol roles such as write-only coordinators, replica-only coordinators, or privileged synchronization coordinators.
- A deployment MAY operate multiple coordinators.
- A group MAY choose one coordinator as its current active write target.
- That write-target choice is operational state, not a distinct coordinator type.

This uniformity requirement is intended to preserve portability between coordinators and reduce protocol complexity.

### 3. Client Role

Clients are responsible for MLS state management and validation.

At a minimum, clients:

- generate MLS KeyPackages and maintain the corresponding private key material
- create and process MLS Proposals, Commits, and Welcomes
- verify coordinator-served KeyPackage publication payloads before use
- track per-group fetch progression using coordinator cursors

Coordinators do not replace MLS validation performed by clients. Coordinator-side checks are defense-in-depth and admission-control measures.

### 4. Per-Group Message Ordering

Each coordinator maintains a monotonic cursor per group.

Properties:

- cursor values are scoped to one group
- cursors define fetch progression and message order within that group on that coordinator
- cursors are pagination and ordering primitives, not canonical message identities
- cursors are coordinator-local values, but per-group scoping is intended to simplify group portability and migration-oriented order preservation compared with a single coordinator-wide cursor space

This document does not define a single global coordinator-wide cursor.

### 5. Fetch Progression

Public fetching remains cursor based.

The fetch interface uses a per-group `afterCursor` progression model.

Properties:

- `afterCursor` is scoped to a specific group
- clients fetch ordered messages after that group cursor
- the returned ordering is defined only within that group on that coordinator

The cursor is not a canonical identity for any message and MUST NOT be treated as such.

### 6. Identity Model for MLS Credentials

`cordn` uses MLS BasicCredential for stable identity mapping.

- Clients MUST use MLS BasicCredential for the initial `cordn` identity model.
- The BasicCredential identity field MUST contain the canonical encoded Nostr public key for the publishing identity.
- The credential identity MUST be treated as stable for the lifetime of the corresponding MLS leaf identity unless replaced through valid MLS credential update rules.

This document intentionally leaves the exact canonical byte encoding as an application-wide convention to be fixed uniformly by implementations. All interoperable implementations MUST use the same encoding.

BasicCredential alone is not sufficient proof of identity ownership. It is only a claimed identity value and MUST be validated using the publication rules below.

### 7. Signed KeyPackage Publication Payload

KeyPackages are published to coordinators as signed publication payloads.

The publication payload MUST bind:

- the publisher identity
- the exact serialized KeyPackage bytes
- any protocol fields required for lookup or freshness

The coordinator MUST store the signed publication payload verbatim.

When serving published KeyPackages, the coordinator MUST return the same signed publication payload, not a coordinator-rewritten substitute representation.

This allows clients to verify the publication payload independently and prevents the coordinator from becoming the sole trust anchor for KeyPackage identity binding.

### 8. Coordinator Validation of KeyPackage Publication

When accepting a KeyPackage publication, a coordinator MUST validate all of the following:

- the authenticated caller identity obtained from the transport matches the signer identity of the publication payload
- the publication payload signature is valid
- the embedded KeyPackage bytes decode successfully
- the BasicCredential identity embedded in the KeyPackage matches the signer identity of the publication payload

If any of these checks fail, the coordinator MUST reject the publication.

Coordinator validation is mandatory even though clients also validate publication payloads later.

In the ContextVM deployment model, the authenticated caller identity is derived from the server request context rather than from unsigned client-supplied input.

### 9. Client Validation of Published KeyPackages

Clients MUST validate coordinator-served KeyPackage publication payloads before using the embedded KeyPackage.

At a minimum, clients MUST verify:

- the publication payload signature is valid
- the embedded KeyPackage bytes match the signed payload contents
- the KeyPackage is structurally valid
- the BasicCredential identity inside the KeyPackage matches the signer identity of the publication payload

Clients MUST reject KeyPackages whose identity binding cannot be verified.

### 10. Trust Model

The `cordn` trust model is layered.

- Coordinators perform admission checks when accepting published data.
- Clients independently verify signed KeyPackage publication payloads on retrieval.
- MLS object validation remains a client responsibility.

This design reduces the amount of trust clients place in coordinators while preserving the coordinator’s role as the storage and distribution service.

### 11. Interoperability Requirements

Implementations MUST agree on all of the following:

- the canonical encoding of the Nostr public key used in BasicCredential identity fields
- the signed KeyPackage publication payload format
- the validation rules that bind the outer signer identity to the inner MLS credential identity
- the per-group cursor semantics used for fetch progression

Implementations MUST reject malformed publication payloads, invalid signatures, malformed KeyPackages, and credential-identity mismatches.

Migration and portability strategies MAY be defined by other `cordn` specifications. They are out of scope for this document and are OPTIONAL unless a future specification states otherwise.

### 12. Rationale

The coordinator model defined here is intentionally minimal.

- Uniform coordinators reduce protocol specialization and operational complexity.
- Per-group cursors provide simple ordered fetching without exposing a global coordinator-wide progression primitive.
- Per-group cursors also make portability easier because a group-specific ordered history can be preserved more naturally if a group later migrates between coordinators.
- Signed KeyPackage publication payloads provide a stronger identity-binding model than trusting coordinator state alone.
- Client-side verification prevents coordinators from being the only enforcement point for KeyPackage identity integrity.

This approach keeps the baseline `cordn` coordinator protocol simple while preserving space for future evolution.
