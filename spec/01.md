# Cordn Group Metadata Extension

- Status: Draft

## Abstract

This document defines the `cordn_group_metadata` MLS GroupContext extension for [`cordn`](package.json). The extension provides a compact, authenticated container for shared group presentation metadata. Its initial purpose is to let members agree on a group name, optional description, optional administrator list, and optional visual identity fields without relying on coordinator-local state alone.

The extension is intentionally optional. Groups may omit it entirely and remain valid. When present, the extension becomes part of the MLS-authenticated group state and is therefore integrity protected by normal MLS Proposal and Commit processing. This allows implementations to expose a consistent view of group metadata to all members while preserving a simple baseline for groups that do not need shared metadata.

The v1 format is intentionally minimal. It supports textual metadata, an optional list of administrator public keys, an optional compact icon string, and an optional image URL. This design leaves room for append-only evolution while avoiding unnecessary coupling to external storage or transport-specific media machinery.

## Specification

### 1. Overview

`cordn_group_metadata` is a custom MLS GroupContext extension.

- It is carried in the MLS group context.
- It is authenticated by MLS group state transitions.
- It is optional at group creation and may be absent.
- When present in a group, all members of that group MUST support its extension type according to MLS extension capability rules.

### 2. Extension Identifier

Implementations MUST use a stable private-use MLS extension type for `cordn_group_metadata`.

This document assigns:

- `cordn_group_metadata = 0xC04D`

This value is in the MLS private-use extension range and MUST be advertised in member capabilities whenever a client claims support for this extension.

### 3. TLS Serialization

The extension payload uses TLS presentation language with MLS variable-length vector encoding conventions.

In TLS notation:

```tls
opaque Name<0..2^16-1>;
opaque Description<0..2^16-1>;
opaque AdminPubkeys<0..2^16-1>;   // concatenated raw 32-byte public keys
opaque Icon<0..2^16-1>;
opaque ImageUrl<0..2^16-1>;

struct {
    uint16 version;
    Name name;
    Description description;
    AdminPubkeys admin_pubkeys;
    Icon icon;
    ImageUrl image_url;
} CordnGroupMetadata;
```

All textual fields MUST be valid UTF-8.

### 4. Versioning

- Version `1` is the initial format defined by this document.
- Version `0` is reserved and MUST be rejected.
- Future versions MUST preserve append-only evolution by adding new fields only at the end of the structure.
- Implementations SHOULD ignore unknown trailing fields from future versions when this can be done safely.

### 5. Field Semantics

#### 5.1 `name`

- UTF-8 encoded group display name.
- MAY be empty.
- Implementations SHOULD apply reasonable local display limits.

#### 5.2 `description`

- UTF-8 encoded group description.
- MAY be empty.

#### 5.3 `admin_pubkeys`

- Encoded as a variable-length byte string containing concatenated raw 32-byte public keys.
- The total byte length MUST be a multiple of 32.
- Duplicate keys MUST NOT appear.
- Omitted semantics are represented by an empty value.

Semantics:

- Empty `admin_pubkeys` means the group is in egalitarian mode.
- In egalitarian mode, all group members have the same administrative level for metadata and group-management actions defined by the application.
- Non-empty `admin_pubkeys` means only the listed members are administrators.

This document does not define how applications map MLS member identities to administrator public keys beyond requiring a stable application-defined public key representation.

#### 5.4 `icon`

- UTF-8 encoded short icon string.
- MAY be empty.
- Intended for a compact visual identifier such as a single letter, glyph, or emoji.
- Producers SHOULD keep this field short.

#### 5.5 `image_url`

- UTF-8 encoded URL or URI reference for a group image.
- MAY be empty.
- This field is advisory presentation metadata.
- Implementations MAY support ordinary remote URLs and inline `data:` URLs.

Security and behavior requirements:

- Clients SHOULD treat referenced content as untrusted remote content.
- Clients MAY decline to fetch the resource.
- Implementations SHOULD apply local policy for acceptable schemes, sizes, and media types.

### 6. Extension Presence and Absence

- A group MAY omit `cordn_group_metadata` entirely.
- If omitted, the group has no shared metadata defined by this extension.
- If present, the encoded payload MUST follow the structure in this document.

### 7. Capability Requirements

- A client that advertises support for `cordn_group_metadata` MUST include `0xC04D` in its MLS supported extensions capabilities.
- A group using this extension MUST ensure that all members support it before adding them.
- A client joining a group that uses this extension MUST verify that it supports the extension before accepting the group state.

### 8. Lifecycle and Updates

- The extension MAY be set during group creation.
- The extension MAY be added, removed, or replaced by MLS `group_context_extensions` proposals and the commits that apply them.
- Because MLS GroupContext extension updates replace the full extension list, senders updating this extension MUST preserve any other GroupContext extensions that remain in use.

When updating the extension payload:

- Implementations MUST serialize the complete `CordnGroupMetadata` structure.
- Empty textual fields represent absence or empty content for their respective semantics.
- Empty `admin_pubkeys` represents egalitarian mode.

### 9. Interoperability Notes

- Implementations MUST reject malformed encodings.
- Implementations MUST reject invalid UTF-8 in textual fields.
- Implementations MUST reject `admin_pubkeys` values whose byte length is not a multiple of 32.
- Implementations SHOULD handle absent or empty optional content consistently across serialization and parsing.

### 10. Rationale

The v1 extension is intentionally minimal.

- `name` and `description` provide immediate product value.
- `admin_pubkeys` supports both egalitarian groups and explicit administrator sets with a single simple encoding.
- `icon` provides a compact visual identity option that does not require network fetches.
- `image_url` provides a flexible image reference without forcing any single storage, privacy, or media-distribution model.

This approach keeps the extension small, authenticated, and easy to implement while preserving room for future append-only evolution.
